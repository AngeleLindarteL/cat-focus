# Feedback Report - 2026-04-13

## Context

This report summarizes the main lessons from the onboarding implementation iteration, the architectural and organizational patterns now being enforced in the repository, and the planning improvements that would reduce rework in future iterations.

## Lessons Learned

### Architecture and boundaries

- The extension works better when popup, options, content, and shared logic are treated as separate surfaces from the start.
- Thin entrypoints are the right default. `src/popup/App.tsx` and `src/options/App.tsx` should remain composition points, not orchestration layers.
- Chrome APIs should stay isolated behind wrappers and repositories in `src/lib`. This keeps views testable and avoids scattering storage and runtime concerns across the UI.
- Storage-backed workflows benefit from a repository pattern early. The onboarding work became more stable once cat data and onboarding state were split into separate repositories.
- Repository ownership clarified responsibility:
  - `CatRepository` owns persisted cat profile data
  - `OnboardingRepository` owns onboarding progression and completion state

### UI and container/view pattern

- The container/view pattern is necessary, not optional, for extension surfaces in this codebase.
- Containers should own:
  - data loading
  - side effects
  - repository calls
  - Chrome integration
  - form submission orchestration
  - step transitions and navigation
- Views should own:
  - markup
  - layout
  - visual hierarchy
  - prop-driven rendering only
- Keeping views free from storage and Chrome logic reduced coupling and made later refactors easier.
- Shared presentational primitives such as `SurfaceCard`, `Stepper`, and `PixelCat` are more reusable when they stay dumb and receive normalized props.

### Folder structure and file organization

- Flat component files created friction once the feature grew. Bundling each component in its own folder is the better default.
- The adopted component rule is:

```text
ComponentName/
  ComponentName.tsx
  index.ts
  interfaces.ts
  constants.ts
  useComponentName.ts
```

- This structure improves local discoverability and gives a clear place for:
  - public props and types
  - local static config
  - view-specific hooks
  - exports
- The same principle also applies to non-UI logic. Repository and storage modules should not mix interfaces, constants, and implementation in one file.
- The adopted logical/backend rule is:

```text
someFeature.ts
someFeature.interfaces.ts
someFeature.constants.ts
```

- This is now the preferred structure for repositories, storage clients, validators, and content-side logic.

### Validation, forms, and state

- `react-hook-form` plus Zod is the right baseline for onboarding forms.
- Validation rules and sanitization need to be explicit in the plan, not implied during implementation.
- Persisted resume behavior should be specified at the repository level, not left to container interpretation.
- Loading persisted values into forms must be part of the expected flow for any resumable onboarding step.

### i18n and copy

- i18n should be considered part of the feature contract from the beginning, not a finishing pass.
- Requiring both English and Spanish upfront prevented hardcoded strings from spreading across containers and views.
- Validation errors, placeholder copy, step labels, and CTA text all need translation coverage as part of the acceptance criteria.

### Verification and implementation flow

- Lint, tests, and build checks caught structure-level issues that were not visible during the first implementation pass.
- The main implementation risk was not feature logic but architectural drift:
  - flat component files
  - mixed responsibilities in `lib`
  - implicit rules not encoded in `AGENTS.md`
- Converting the learned rules into repo instructions immediately after implementation reduced the chance of repeating the same mistakes.

## Patterns Now Enforced

### Project architecture

- Browser-extension-first architecture
- Thin surface entrypoints
- Clear separation between popup, options, content, modules, components, and lib
- Repositories as the persistence abstraction for storage-backed features
- Chrome API isolation under `src/lib`

### UI and React patterns

- Mandatory container/view pattern for renderable surfaces
- Shared components remain presentational
- Views accept normalized props and callbacks
- Containers orchestrate behavior, persistence, and side effects
- Hooks are used for reusable or stateful logic, but remain scoped to their component or module context

### File and folder patterns

- React components use a bundled folder structure
- Non-UI logical modules split constants, interfaces, and implementation into separate files
- Local constants should live in `constants.ts`, not inline in implementation files
- Public exports should flow through `index.ts` barrels for component folders

## What Can Be Improved At Planning Level

### 1. Lock architecture rules before implementation starts

- The first plan defined feature behavior well, but it did not fully lock the code organization rules.
- This caused rework when component bundling, repository file splits, and constants placement were clarified later.
- Future plans should explicitly state:
  - component folder shape
  - logical/backend file split rules
  - whether constants must be extracted
  - expected public import surface

### 2. Specify the persistence abstraction up front

- The original implementation direction assumed storage wrappers; later the requirement changed to repositories.
- That change affected types, flow ownership, and file organization.
- Future plans should decide early:
  - direct storage client vs repository pattern
  - repository boundaries
  - which repository owns each piece of state

### 3. Make resumability rules decision-complete

- “User can leave and come back” was not enough by itself.
- The plan should define:
  - what gets persisted
  - what gets reloaded
  - whether the user resumes the exact step or restarts from step 1
  - how revisiting prior steps should prefill data
- These rules should be attached to the repository contract, not left to UI interpretation.

### 4. Define state-machine behavior more precisely

- The onboarding had a 3-step contract, but only step 1 was fully implemented.
- That required later decisions about placeholder behavior and completion semantics.
- Future plans should always include:
  - valid step values
  - normalization rules for invalid/missing values
  - behavior for unimplemented future steps
  - completion transition rules

### 5. Include i18n and validation requirements as first-class acceptance criteria

- Translation and validation were known requirements, but they should have been framed as concrete deliverables in the initial spec.
- Future plans should explicitly list:
  - required locales
  - which strings must be localized
  - required validation and sanitization rules
  - expected input error states

### 6. Capture UI architecture, not only UI behavior

- The initial plan described what the UI should do, but not enough about how the UI should be composed.
- Future UI plans should include:
  - container names and responsibilities
  - view names and responsibilities
  - shared component candidates
  - whether a component is dumb, stateful, or repository-aware

### 7. Define done criteria for verification

- Verification passed after refactoring, but the plan would be stronger if it required the following from the start:
  - lint passes
  - unit tests added for repositories and validation
  - UI behavior tests added for gating and progression
  - production build passes

## Recommended Planning Checklist For Next Iterations

- Confirm the persistence abstraction before coding starts.
- Confirm repository ownership for every persisted state slice.
- Lock resumability behavior and step state normalization rules.
- Specify component and logical-module folder/file structure in the implementation section.
- Define container/view splits explicitly for every renderable surface.
- Define locales, validation, sanitization, and error-state requirements explicitly.
- Define placeholder behavior for incomplete future steps if the flow contract is larger than the implemented scope.
- Require lint, tests, and build success as part of the acceptance criteria.

## Final Takeaway

The main lesson from this onboarding iteration is that behavior-level planning was not enough on its own. The next iteration will move faster if the plan is both product-complete and architecture-complete: persistence model, ownership boundaries, folder conventions, container/view splits, validation rules, and verification criteria should all be decided before implementation starts.
